peal network · engineering

How Peal is built, crate by crate

A reveal-later encryption network is five Rust crates, one wasm bridge, a TypeScript SDK, and a browser app. This is the whole thing wired together: what each part does, why it exists, and how a single secret travels from a browser tab to a committee and back.

grounded in the source · phases 0-8 green

01

the problem

Holding time is not a thing one machine can do

Encryption keeps a secret from outsiders. It says nothing about when the holder of the key chooses to look. "Encrypted" only ever means "someone has the key," and that someone can always peek early. So there is no ordinary way to send a message that literally cannot be opened until a set moment, without trusting a person or a server not to look.

Peal turns "cannot be read until later" into a property of the message itself. You seal a payload to a committee of operators. No single operator can open it. Only when a cue fires, a wall-clock time or a chain height, does the whole batch open at once, reconstructed from a threshold of operator shares. Nothing is readable early, not even by the operators, as long as fewer than the threshold collude.

The scheme underneath is commonware's batched threshold encryption over the BLS12-381 pairing curve. Peal is the plumbing that turns that scheme into a running network: a coordinator, operator nodes, a client SDK that seals in the browser, and an explorer that watches it happen. The tagline in the code is literal: seal now. reveal on cue.

02

the shape

Six programs, one crypto core

Everything is a workspace: five Rust crates plus a wasm target, a TypeScript SDK, and a browser app. The rule that keeps the system honest is that exactly one crate is allowed to touch elliptic-curve group elements. Everything else moves opaque bytes around.

programwhat it iswhat it does
bte-cryptothe crypto coreThe only crate touching group elements. Wraps commonware's simple-bte (pinned at rev 147a0878), adds wire formats, payload caps, and per-slot validity.
bte-coordinatorthe serveraxum /v0 REST API over a single sqlite connection. Runs the engine that freezes batches, gathers shares, and builds reveals. Holds no key material.

03

the core

bte-crypto, the one crate that touches the curve

All pairing math, group FFTs, and the batched-threshold scheme itself live in simple-bte. bte-crypto is a thin, opinionated surface over it: wire types prefixed with the magic BTE0, a 4096-byte payload cap, and an API shaped for the coordinator, node, and SDK. Two feature flags split the surface. seal-only is just wire types plus seal, which is exactly what the wasm build compiles. full adds the operator and recovery side.

The whole cryptographic protocol reduces to seven functions. Read them in order and you have the entire lifecycle.

crates/bte-crypto/src/lib.rs 
// client side, browser-safe (seal-only feature)
pub fn seal(params: &PublicParams, payload: &[u8], rng) -> SealedCiphertext

// dealer side, run once per committee (full feature)
pub fn ceremony(n, t, b, rng) -> (PublicParams, Vec<OperatorSecret>)

// operator side, once per frozen batch
pub fn partial(secret: &OperatorSecret, batch: &[CtHeader]) -> Share
pub fn verify_share(params, batch: &[CtHeader], share: &Share) -> bool

// recovery side, run by the coordinator at reveal
pub fn pre_decrypt(rk: &RecoveryKey, batch) -> PrecomputedCrossTerms
pub fn combine(shares: &[Share]) -> CombinedShare
pub fn finalize(rk, pre, combined, batch) -> Vec<RecoveredPayload>

What a sealed ciphertext actually is

A SealedCiphertext is three fields: a 48-byte G1 point ct0 (the KEM header, written [k]_1), a 16-byte key mask ct1, and a payload-length keystream body ct2. That first 48-byte point is the only thing an operator needs to do its job. It never sees the body. The payload path is the Fujisaki-Okamoto transform (simple_bte::bte::fo), used here as a CCA-secure data-encapsulation layer, not the G_T / Schnorr variant.

Why B is baked in, and why there are dummies

The scheme is batched. The committee's setup fixes a batch size B (64 in this deployment) into the punctured powers-of-tau, the FFT domain, and the encryption key. Every batch must be exactly B slots. That single fact explains a design choice that looks strange at first: when a real batch has fewer than B ciphertexts, the coordinator pads it with self-sealed dummies. Each dummy is a real FO ciphertext sealing BTE_DUMMY_V0: plus 16 random bytes. Operators do real work on them, they are committed in the merkle root with everything else, and they are indistinguishable on the wire except for their position. Padding is not decoration, it is what makes the fixed-B math close.

One share for a whole batch, 48 bytes

An operator's partial decryption for an entire frozen batch is a single G1 point, independent of B. It is one multi-scalar multiplication, pd_j = sum_i sigma_j^i · ct_{i,0}, the operator's Shamir shares against the batch's headers. So the network cost of decrypting 64 sealed messages is one 48-byte upload per operator, not 64.

Public verifiability is the load-bearing property.

Anyone with the public params can check a share with a pairing equation, e(pd_j, g_2) == prod_i e(ct_{i,0}, v_j^i), and it returns false rather than panicking on any mismatch. This is what lets the coordinator, which holds no secrets, reject a lying operator without being able to decrypt anything itself. A byzantine node's share is flagged and never counted toward the threshold.

04

the lifecycle

Following one secret from tab to reveal

Here is the whole path, step by step, with the real function at each hop.

1. A condition is created

The SDK's condition({in: 60}) or {atBlock: {chainId, height}} posts to /v0/conditions. A condition is the cue: fire at a wall-clock time, or when a chain reaches a height. It starts in status pending.

2. The secret is sealed in the browser

This is the part people miss: encryption happens client-side, in wasm, before anything leaves the tab. BteClient.seal fetches the committee's public params once, parses them inside wasm (which subgroup-checks every point), and calls params.seal(bytes). Only the ciphertext is posted to /v0/ciphertexts. The plaintext never touches the network.

packages/sdk/src/index.ts 
const { params } = await this.ensureParams();
const sealed = params.seal(bytes);         // wasm, in-tab
await this.request('/v0/ciphertexts', {  // only ciphertext leaves
  method: 'POST',
  body: JSON.stringify({ condition_id, sealed_blob_b64: sealedB64 }),
});

The SDK also verifies that the params digest it computed in wasm matches the digest the coordinator advertised, so a coordinator that served inconsistent params is caught before you seal to it.

3. The cue fires and the batch freezes

The coordinator's engine ticks on an interval (tick in engine.rs, every 500ms in production). On each tick it finds conditions whose time has passed, or polls the referenced chain's head via eth_blockNumber for block conditions. When one is due it freezes:

  • Real ciphertexts are sorted by ct_hash, so their positions are a pure function of the ciphertext set, not of arrival order. Nobody can buy slot zero.
  • The tail is padded with dummies up to a multiple of B.
  • Positions are assigned, batch rows are written, status becomes frozen.
  • Pipelining: the coordinator immediately runs pre_decrypt for each batch. The cross-terms depend only on the ciphertexts and public params, not on any share, so this expensive FFT step runs before a single operator has responded. The result is cached in memory keyed by batch id.

4. Operators do their one job

Each bte-node is a stateless poller. It hits GET /v0/work?operator=N, gets back the packed 48-byte headers for every frozen batch it has not answered, computes partial(secret, headers), and posts the 48-byte share to /v0/shares. Then it sleeps and does it again. It is safe to restart at any point, because it holds no state beyond its encrypted keystore.

crates/bte-node/src/main.rs 
let share = if cfg.byzantine {
    random_invalid_share(cfg.operator_id)   // dev-only, gated behind BTE_DEV=1
} else {
    partial(secret, &headers)?              // one MSM, 48 bytes out
};

5. Shares are verified at the door

When a share lands, the coordinator runs verify_share inline before storing it. A share that fails the pairing check is stored flagged and never counts toward the threshold t. This is the whole byzantine-tolerance story: a malicious operator cannot poison a reveal, it can only decline to help. A batch finalizes the moment it has t verified shares.

6. Finalize and reveal

Once a batch has t good shares, the engine combines them by Lagrange interpolation at x=0 (combine), then finalize runs B pairings against the cached cross-terms and opens every slot. Finalize also re-derives each slot's randomness and checks [k_i]_1 == ct0_i, so one mauled ciphertext is marked invalid instead of corrupting its neighbours. When every batch in a condition is finalized, the coordinator assembles the reveal, computes a merkle root over (position, payload) leaves, and flips the condition to revealed.

Liveness has a real boundary.

The guarantee is "opens when the cue fires" only if at least t operators are honest and online. If a frozen batch waits past REVEAL_TIMEOUT_SECS without reaching t, it is marked stalled rather than silently hanging. Stalled conditions still finalize if late shares arrive. The reveal endpoint /v0/reveals/:id returns 404 until the reveal exists, enforced by a test that greps the database for plaintext before reveal.

05

the server

Inside the coordinator

The coordinator is deliberately boring where it can be and careful where it must be. It is axum over a single Mutex<Connection> to sqlite. The entire public surface is nine routes under /v0: conditions, ciphertexts, work, shares, reveals, committees, and health. CORS is hand-rolled middleware (no new dependency), and there is a per-IP token-bucket rate limiter in front of everything.

The one piece of real machinery is the engine, a state machine over four condition states:

statemeaningtransition
pendingaccepting ciphertexts, cue not yet mettime passed or block reached → freeze
frozenpadded, positioned, pre-decrypted, awaiting sharesevery batch hits t verified → reveal
revealedpayloads and merkle root publishedterminal
stalledwaited past timeout without t sharesstill finalizes if late shares arrive

A subtle correctness detail: the pre-decrypt cross-terms live in an in-memory map, not the database. If the coordinator restarts between freeze and finalize, that cache is empty, so finalize_batch detects the miss and recomputes the cross-terms on the fly before combining. The system survives a restart at any point in a batch's life without losing a reveal.

What the coordinator cannot do:

It never holds tau, and it never holds an operator's shares. It stores ciphertexts, routes headers to operators, verifies and counts shares, and runs the public recovery math once t shares exist. It cannot decrypt anything early on its own, because decryption fundamentally requires t operator shares it does not have. It is trusted for liveness and for honest ordering, not for confidentiality.

06

the operator

Keystores and the byzantine switch

An operator's secret is Shamir shares of tau^1..tau^B. At rest it lives in an encrypted keystore: argon2id derives a key from a passphrase, ChaCha20-Poly1305 encrypts the wire bytes. The struct deliberately has no Debug impl, and no code path logs key material. Cloud deploys that cannot share a volume can inject the encrypted keystore JSON through an env var instead of a file.

crates/bte-node/src/keystore.rs 
pub fn seal_keystore(secret: &OperatorSecret, passphrase: &str) -> KeystoreFile {
    let key = derive_key(passphrase, &salt)?;    // argon2id
    let cipher = ChaCha20Poly1305::new((&key).into());
    // plaintext = OperatorSecret BTE_WIRE_V0 bytes, never touches a log
}

The --byzantine flag posts a wire-valid but cryptographically wrong share (a random G1 point). It refuses to start unless BTE_DEV=1. This exists so the test suite can prove the coordinator flags and rejects bad shares, and it is exactly what the demo-byzantine run exercises: node 5 is killed mid-flow, node 2 lies, the reveal still completes on three honest shares.

07

the bridge

How the browser seals without a backend round trip

bte-wasm is a small wasm-bindgen surface over bte-crypto. It exposes a Params object you construct once (parsing subgroup-checks every point) and reuse, plus seal and a content-address helper. The SDK's build step inlines the compiled wasm as base64, so there is no bundler config to get right in Vite, Next, or Node. The first call lazily initializes the module from the inlined bytes.

The getrandom wiring is the fiddly part and worth naming, because it bites everyone who tries this. The wasm build needs a cargo cfg selecting the wasm_js backend, and the workspace pins its RNG to the exact version simple-bte's API expects (ark-std 0.6 re-exports rand 0.8). Downstream code uses bte_crypto::os_rng() rather than depending on rand directly, so the two rand versions in the tree can never silently diverge.

On top of the base scheme, the explorer adds an optional AES-128-GCM layer for private seals, with the symmetric key living only in the share-link URL fragment (which never reaches a server). That is how a "time capsule" link stays unreadable even to someone watching the coordinator: the committee opens the outer capsule, but the inner content needs the key that only travelled in the # fragment.

08

the proof

Making a reveal checkable by a stranger

The reveal is not "trust the coordinator's JSON." Every reveal carries a merkle root over its slots, leaf = sha256(position_le_u32 || payload), parents sha256(left || right), odd nodes promoted. The SDK reimplements the exact same tree in computeRevealRoot, so a client can recompute the root from the payloads it received and confirm the coordinator did not reshuffle or edit anything.

Optionally, that root is anchored onchain. The anchor helpers are dependency-free: calldata is hand-encoded for fixed bytes32 arguments, reads go through a raw eth_call, hashing through WebCrypto, so the SDK pulls in no chain library. verifyAnchor closes the loop: it recomputes the root from the revealed payloads, reads the root the key-holder published to BteAnchor.sol, and returns matches only when the onchain root, the recomputed root, and the coordinator's stated root all agree.

packages/sdk/src/anchor.ts 
matches: onchainRoot === recomputedRoot && recomputedRoot === reveal.merkleRoot

The coordinator itself stays chain-free. The reveal-root transaction is sent by a key-holder script, not the server, so the coordinator never needs a funded key or an RPC to a mainnet.

09

trade-offs

The decisions worth defending

One crate touches the curve, on purpose.

Keeping every group operation inside bte-crypto means the coordinator, node, and SDK all move opaque, wire-checked bytes. A bug in the pairing math has exactly one place to be. It also made the wasm surface trivial: the browser links the same seal the server does.

FO as the payload path, not G_T.

The Fujisaki-Okamoto transform gives CCA security and lets a payload be arbitrary bytes up to the cap, rather than a group element. The cost is the per-slot re-derivation check at finalize, which is also what buys the "one mauled ciphertext cannot poison the batch" property. That was a deliberate trade recorded in the deviations notes.

The coordinator is trusted for liveness, not secrecy.

A malicious coordinator can refuse service, stall a reveal, or lie about which ciphertexts exist, which is exactly why positions are content-derived and reveals are merkle-committed. It cannot read a sealed payload early, because that needs t operator shares it does not hold.

The real caveat: v0 uses a single trusted dealer.

The ceremony that creates a committee samples tau in one process, Shamir-deals the shares, and drops tau. While it ran, that one process knew tau, and tau can decrypt anything sealed to the committee. The code says so out loud: the ceremony prints "this process was the trusted dealer. tau is gone." Peal's whole value is trust-minimization, so this single-dealer setup is treated as a launch blocker, not a footnote. The fix is a multi-party setup ceremony (secure if one contributor is honest) or full distributed key generation. Until then, the honest use cases are low-stakes and non-adversarial.

10

results and next

Where it stands, and what finishes the picture

Phases 0 through 8 are green: the crypto crate's property tests, the coordinator's invariant tests (including the byzantine-share and stall-recovery cases), a live end-to-end run against a five-node Docker network, the SDK's vitest suite, and the anchored-reveal test against a local chain. Measured on an M-series laptop, single process: seal about 416µs, one operator partial about 1.16ms, share verification about 12ms, and full batch recovery (pre-decrypt plus finalize for B=64) about 282ms, end-to-end recovery about 432ms. Those track the paper's published scale. The committee runs at n=5, t=3, B=64: three honest, online operators are enough to open a batch; two colluding operators learn nothing.

Two things turn this from a working system into the network it claims to be. First, remove the single dealer, so "no one can peek early" is verifiable by a stranger rather than resting on one machine's promise to delete tau. Second, put it under one real workload with real stakes, the sealed-bid or sealed-order-flow case, where being unreadable-until-the-cue is worth paying for. The cryptography is done. The trust boundary and the first serious user are what remain.
